1 Validation HTML

Le HTML doit être valide. Pas "à peu près". Zéro erreur, zéro warning non justifié. Un document invalide casse le parsing, l'accessibilité, le SEO, et le rendu cross-browser.

W3C Markup Validation Service

validator.w3.org

Validateur officiel du W3C. Accepte une URL, un upload, ou du code collé. Utilisez "validate by direct input" pour les pages locales.

  • ✓ 0 erreur, 0 warning
  • ⚠ 0 erreur, warnings mineurs
  • ✗ Toute erreur = à corriger

HTMLHint (CLI)

htmlhint.com

Linter HTML en ligne de commande. Vérifie les attributs manquants, les balises non fermées, les id dupliqués. 

npx htmlhint "**/*.html"

2 Accessibilité

WCAG 2.1 niveau AA minimum. L'accessibilité n'est pas un bonus — c'est un critère d'évaluation.

WAVE — Web Accessibility Evaluation Tool

wave.webaim.org

Extension navigateur (Chrome/Firefox) ou URL directe. Affiche erreurs, alertes, et structure ARIA directement sur la page.

  • ✓ 0 erreur
  • ⚠ Alertes = à vérifier manuellement
  • ✗ Toute erreur = à corriger

axe DevTools

deque.com/axe/devtools

Extension Chrome. Audit WCAG automatisé dans les DevTools. Plus précis que WAVE sur certains cas ARIA. Utilisez les deux — ils ne détectent pas les mêmes problèmes.

  • ✓ 0 violation "critical" et "serious"
  • ⚠ Violations "moderate" = à traiter si possible

WebAIM Contrast Checker

webaim.org/resources/contrastchecker

Vérification rapide du ratio de contraste entre deux couleurs.

  • ✓ ≥ 4.5:1 texte normal, ≥ 3:1 texte large (≥18px bold ou ≥24px)
  • ✗ En dessous = non conforme AA

Tests manuels obligatoires

TestComment
Navigation clavierTab sur toute la page. Chaque élément interactif doit être atteignable et avoir un :focus-visible visible
Skip linkPremier Tab = lien "Aller au contenu" (ou équivalent)
Zoom 200%Aucun contenu tronqué, pas de scroll horizontal
Lecteur d'écranAu minimum : tester landmarks, headings, et formulaires avec VoiceOver (macOS) ou NVDA (Windows)

3 Performance

Vanilla HTML/CSS/JS = aucune raison d'avoir un Lighthouse < 90.

Lighthouse (Chrome DevTools)

DevTools → Lighthouse → cocher Performance, Accessibility, Best Practices, SEO

Exécutez en mode mobile (throttled), en navigation privée (pas d'extensions). Lancez au moins 3 fois, gardez le score médian.

  • ✓ Perf ≥ 90 · A11y ≥ 95 · BP ≥ 90 · SEO ≥ 90
  • ⚠ Performance 70–89 = optimisable
  • ✗ Tout score < 70 = problème sérieux

PageSpeed Insights

pagespeed.web.dev

Lighthouse distant (données réelles + simulées). Utile si le site est déployé.

5 Internationalisation (obligatoire)

Chaque projet doit livrer un système de traduction fonctionnel, activable en incluant deux balises <script> en fin de document. Pas de librairie externe. Pas de rechargement de page. Vanilla JS.

Architecture requise

Le système repose sur deux fichiers distincts, avec des responsabilités séparées. Les noms de fichiers, le format de stockage, et le mécanisme de sélection de langue sont libres — mais la séparation en deux scripts et le contrat d'intégration sont imposés.

ScriptResponsabilitéContenu type
Script 1 — stockage des traductions Déclare toutes les chaînes traduites pour toutes les langues supportées. Ne contient aucune logique de comportement. Un objet JS (ou JSON chargé) structuré par langue et par clé. Exemple : { "fr": { "nav.home": "Accueil" }, "nl": { "nav.home": "Home" } }
Script 2 — comportement de traduction Lit les traductions du script 1, détecte ou reçoit la langue cible, et applique les traductions au DOM. Gère le sélecteur de langue. Parcourt les éléments marqués (par attribut data-i18n, par convention de classe, ou toute autre méthode), remplace le contenu texte, met à jour l'attribut lang sur <html>, et persiste le choix utilisateur.

Contrat d'intégration

Depuis n'importe quelle page HTML du projet, l'ajout de ces deux lignes avant </body> doit suffire à activer la traduction, sans autre modification du document : 

<script src="js/i18n-data.js"></script>
<script src="js/i18n.js"></script>

Les noms de fichiers ci-dessus sont un exemple. Chaque groupe choisit ses noms. L'essentiel : deux includes, et ça fonctionne.

Critères d'évaluation

PointAttendu
Activation par inclusion Deux <script> en fin de document = traduction opérationnelle. Pas de configuration supplémentaire
Séparation données / logique Les traductions vivent dans un fichier, le comportement dans un autre. Ajouter une langue = modifier uniquement le fichier de données
Sélecteur de langue Un contrôle accessible (bouton, <select>, ou liens) permet de changer de langue. Le choix persiste (session ou localStorage)
Mise à jour du DOM Le changement de langue s'applique sans rechargement de page. L'attribut lang sur <html> est mis à jour
Minimum 2 langues Le site doit être traduit dans au moins 2 langues (français + une autre pertinente pour le projet)
Attribut lang <html lang="…"> reflète toujours la langue active
Passages dans une autre langue Si du contenu reste dans une langue fixe (nom propre, citation), il porte un lang explicite
Pas de contenu "oublié" Tous les textes visibles sont traduits. Les alt, title, aria-label, et placeholder aussi
Fallback Si une clé est absente pour une langue, le système affiche la langue par défaut, pas une clé brute
Progressive enhancement Sans JS, le contenu s'affiche dans la langue par défaut du HTML. Le site reste lisible

Balisage HTML

PointAttendu
EncodageUTF-8 partout. Pas d'entités HTML inutiles — écrire é, pas &eacute;
DatesFormat européen (jj/mm/aaaa) ou ISO. Pas de format US
hreflangSi les langues ont des URLs distinctes, déclarer les alternatives via <link rel="alternate" hreflang="…">

6 CSS & layout

Deux approches de layout existent. Choisissez-en une et tenez-la. Les deux sont valides. Elles ne se mélangent pas bien.

Choisissez votre approche

A — Intrinsic design

Le layout s'adapte au contenu et à l'espace disponible, sans breakpoints. Les composants décident eux-mêmes comment se comporter.

Outils CSS :

clamp(), min(), max() pour les tailles fluides. grid avec auto-fit/auto-fill + minmax(). flex-wrap avec des flex-basis en ch ou rem. Container queries (@container) pour les composants qui réagissent à leur conteneur. Unités viewport fluides (vi, vb, dvh).

Quand choisir : sites à contenu variable (articles, galeries, catalogues), composants réutilisables, quand les breakpoints arbitraires cassent plus qu'ils n'aident.

B — Responsive design (breakpoints)

Le layout change à des largeurs de viewport définies. Mobile-first : on part du petit écran, on ajoute des règles via min-width.

Outils CSS :

@media (min-width: …) avec des breakpoints explicites. Grilles et flex reconfigurés par breakpoint. Unités relatives (rem, em, %).

Quand choisir : layouts très contrôlés (landing pages), quand le design impose des changements structurels nets entre mobile et desktop.

7 JavaScript

JS est un outil de progressive enhancement. Il ajoute du confort, pas de l'accès. Le contenu doit être lisible sans JS.

ESLint (CLI)

eslint.org

npx eslint "**/*.js"

Vérifications

PointAttendu
Progressive enhancementContenu lisible et navigable sans JS
Chargementdefer ou type="module". Jamais de <script> bloquant dans <head>
"use strict"Présent (ou modules ES qui l'impliquent)
ConsoleAucun console.log oublié. Aucune erreur dans la console
Event listenersPas de onclick inline. addEventListener uniquement
Pas de librairies lourdesPas de jQuery, pas de framework. Vanilla JS ou micro-librairie justifiée (< 10 KB)
Pas de localStorage abusifPréférence thème/langue = OK. Données personnelles = NON
